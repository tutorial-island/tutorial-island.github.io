Implementing a Simple Text-Based Adventure Game in Rust

Oct 27, 2023· Tutorial Island

Introduction

Today, we’re diving into Rust by building a simple text-based adventure game! This game will allow the user to navigate through a story by making choices, and the story will progress based on those choices. This is a great way to understand fundamental Rust concepts in a fun and engaging way.

Explanation

Creating a text-based adventure game presents several challenges:

  1. User Input: We need to read input from the user, allowing them to choose actions or answer questions. Rust’s standard input/output library (std::io) will be crucial here.
  2. Game State: We need to maintain the current state of the game, such as the player’s location, inventory, and health. This requires using appropriate data structures like structs and enums.
  3. Conditional Logic: The game’s narrative will branch based on the user’s choices. if, else if, and match statements will be essential for implementing this branching logic.
  4. Output: We need to display information to the user, telling them about their surroundings, the consequences of their actions, and the current game state. We’ll use println! for this.

We will use these Rust features to tackle these challenges:

  • std::io::stdin for reading user input.
  • String to store user input and story text.
  • struct to represent the game state.
  • enum to represent choices/options.
  • match for complex conditional branching.
  • println! for outputting text to the console.

Requirements

  • Functional Requirements:

    • The game should present the user with a starting scenario.
    • The game should offer the user choices based on the current scenario.
    • The game should update the game state based on the user’s choices.
    • The game should continue until a winning or losing condition is met.
    • The game should handle invalid user input gracefully.

  • Non-Functional Requirements:

    • The code should be well-structured and easy to understand.
    • The game should be reasonably engaging. (Although gameplay is secondary to the learning objective).

Solutions

fn get_user_input() -> String {
    use std::io::{self, Write};

    print!("> ");
    io::stdout().flush().unwrap(); // Ensure prompt is displayed immediately

    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("Failed to read line");
    input.trim().to_lowercase()
}

#[test]
fn test_get_user_input_not_empty() {
    // This test is difficult to automate without mocking stdin,
    // But we can at least ensure the function exists and returns a String.
    let input = get_user_input();
    assert!(input.len() >= 0);
}

#[test]
fn test_get_user_input_lowercase() {
    // This test is difficult to automate without mocking stdin,
    // But we can at least ensure the function exists and returns a String.
    // and that it returns a lowercase string after inputting uppercase strings
    // the proper way to test would be to mock the stdin to give a deterministic behaviour

    assert_eq!(input, input.to_lowercase());
}

This function, get_user_input(), reads a line of text from the standard input. It displays a prompt (> ) to the user, flushes the output buffer to ensure the prompt is visible, and then reads the input into a String. The .trim() method removes any leading or trailing whitespace, and .to_lowercase() normalizes the input for easier comparison later. Flushing the output stream is important to ensure the prompt > is displayed before the program waits for input.

#[derive(Debug, PartialEq)]
enum GameState {
    Start,
    Forest,
    Cave,
    Treasure,
    GameOver,
}

#[test]
fn test_game_state_enum() {
    assert_eq!(GameState::Start, GameState::Start);
    assert_ne!(GameState::Forest, GameState::Cave);
}

This defines an enum called GameState. Enums are useful for representing a fixed set of possible states. In this case, the game can be in the Start, Forest, Cave, Treasure, or GameOver state. The #[derive(Debug, PartialEq)] annotation allows us to easily print the enum’s value for debugging and compare enum values for equality in tests.

fn handle_start(input: &str) -> GameState {
    match input {
        "start" => {
            println!("You find yourself at the entrance to a dark forest.");
            GameState::Forest
        }
        _ => {
            println!("Invalid command. Type 'start' to begin.");
            GameState::Start
        }
    }
}

#[test]
fn test_handle_start_start() {
    assert_eq!(handle_start("start"), GameState::Forest);
}

#[test]
fn test_handle_start_invalid() {
    assert_eq!(handle_start("invalid"), GameState::Start);
}

The handle_start function processes the user’s input when the game is in the Start state. It uses a match statement to determine the appropriate action. If the user types “start”, the function prints a message and transitions the game to the Forest state. Otherwise, it prints an error message and remains in the Start state. The match statement ensures that all possible input cases are handled.

fn handle_forest(input: &str) -> GameState {
    match input {
        "go left" => {
            println!("You enter a dark cave.");
            GameState::Cave
        }
        "go right" => {
            println!("You are lost in the forest and die of starvation. GAME OVER!");
            GameState::GameOver
        }
        _ => {
            println!("Invalid command. Type 'go left' or 'go right'.");
            GameState::Forest
        }
    }
}

#[test]
fn test_handle_forest_left() {
    assert_eq!(handle_forest("go left"), GameState::Cave);
}

#[test]
fn test_handle_forest_right() {
    assert_eq!(handle_forest("go right"), GameState::GameOver);
}

Similar to handle_start, handle_forest handles input when the game is in the Forest state. Based on whether the user chooses “go left” or “go right”, the game transitions to either the Cave state or the GameOver state. This demonstrates simple branching logic.

fn handle_cave(input: &str) -> GameState {
    match input {
        "search" => {
            println!("You find a hidden treasure! You win!");
            GameState::Treasure
        }
        "leave" => {
            println!("You leave the cave and get lost in the forest. GAME OVER!");
            GameState::GameOver
        }
        _ => {
            println!("Invalid command. Type 'search' or 'leave'.");
            GameState::Cave
        }
    }
}

#[test]
fn test_handle_cave_search() {
    assert_eq!(handle_cave("search"), GameState::Treasure);
}

#[test]
fn test_handle_cave_leave() {
    assert_eq!(handle_cave("leave"), GameState::GameOver);
}

The handle_cave function handles input when the player is in the cave. Choosing to “search” wins the game, while choosing to “leave” ends the game.

Thoughts

Now let’s combine all the pieces to create a complete, runnable game. We’ll use a loop to keep the game running until the player wins or loses, and a match statement to dispatch input to the appropriate handler function based on the current game state.

use std::io::{self, Write};

#[derive(Debug, PartialEq)]
enum GameState {
    Start,
    Forest,
    Cave,
    Treasure,
    GameOver,
}

fn get_user_input() -> String {
    print!("> ");
    io::stdout().flush().unwrap();

    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("Failed to read line");
    input.trim().to_lowercase()
}

fn handle_start(input: &str) -> GameState {
    match input {
        "start" => {
            println!("You find yourself at the entrance to a dark forest.");
            GameState::Forest
        }
        _ => {
            println!("Invalid command. Type 'start' to begin.");
            GameState::Start
        }
    }
}

fn handle_forest(input: &str) -> GameState {
    match input {
        "go left" => {
            println!("You enter a dark cave.");
            GameState::Cave
        }
        "go right" => {
            println!("You are lost in the forest and die of starvation. GAME OVER!");
            GameState::GameOver
        }
        _ => {
            println!("Invalid command. Type 'go left' or 'go right'.");

        }
    }
}

fn handle_cave(input: &str) -> GameState {
    match input {
        "search" => {
            println!("You find a hidden treasure! You win!");
            GameState::Treasure
        }
        "leave" => {
            println!("You leave the cave and get lost in the forest. GAME OVER!");
            GameState::GameOver
        }
        _ => {
            println!("Invalid command. Type 'search' or 'leave'.");
            GameState::Cave
        }
    }
}

fn main() {
    let mut game_state = GameState::Start;

    println!("Welcome to the Text Adventure Game!");

    loop {
        match game_state {
            GameState::Start => {
                let input = get_user_input();
                game_state = handle_start(&input);
            }
            GameState::Forest => {
                let input = get_user_input();
                game_state = handle_forest(&input);
            }
            GameState::Cave => {
                let input = get_user_input();
                game_state = handle_cave(&input);
            }
            GameState::Treasure => {
                println!("Congratulations! You found the treasure and won the game!");
                break;
            }
            GameState::GameOver => {
                println!("Game Over!");
                break;
            }
        }
    }
}
